# OpenChat 代码合规审查技能

## 概述

本技能实现一套完整的代码合规审查工作流，依据两份规则文档对项目所有源码文件进行系统化审查，
发现不符合规则的代码后自动修复，然后生成带时间戳的锁定审查文档，最后完成 Git 提交与推送。

## 规则来源

审查依据以下两份文档中的 **全部规则**：

1. **全局规则文档**：`C:\Users\Study_Superior\.Codex\AGENTS.md`
2. **项目规则文档**：项目根目录下的 `AGENTS.md`

在审查开始前，**必须**先读取这两份文档获取完整规则集。当两份文档存在冲突时，
项目级 AGENTS.md 的规则优先。

## 审查报告存放位置

所有审查文档统一存放在项目根目录下的 `audit-reports/` 文件夹中。
每次审查都会在此目录下生成一个带时间戳的唯一文件。

## 完整审查工作流

### 第一阶段：准备与增量判断

1. **读取规则文档**
   - 读取 `C:\Users\Study_Superior\.Codex\AGENTS.md`（全局规则）
   - 读取项目根目录下的 `AGENTS.md`（项目规则）
   - 将两份文档的所有规则提取并合并为完整的审查检查清单

2. **检查上一轮审查文档**
   - 扫描 `audit-reports/` 目录，按文件名中的时间戳找到最新的一份审查文档
   - 如果存在上一轮文档，读取其内容，提取以下信息：
     - 已审查过的文件列表及其审查结果
     - 记录的 Git 版本号
   - 使用 `git diff <上次版本号>..HEAD --name-only` 获取自上次审查以来变更过的文件列表
   - **跳过规则**：上一轮文档中标记为"通过"且此后未发生任何修改的文件，本轮不重复审查
   - **必须审查**：自上次审查后有任何变更的文件、新增的文件、上次标记为"不通过"的文件

   如果 `audit-reports/` 目录不存在或为空，则视为初始审查，扫描所有源码文件。

3. **确定审查范围**
   - 列出本轮需要审查的文件清单
   - 告知用户本轮审查范围（共多少个文件，其中多少是增量的）

### 第二阶段：逐文件审查

对每个待审查文件，按以下维度逐项检查。检查项源自两份 AGENTS.md 的全部规则：

#### A. 架构与模块化（项目 AGENTS.md 第 1-3 条）
- [ ] `main.ts` 是否仅包含注册、初始化和生命周期管理？是否包含具体业务逻辑？
- [ ] 命令逻辑是否分离到 `commands/` 目录？
- [ ] UI 组件（Modal、View、SettingTab）是否在独立文件中？
- [ ] 业务逻辑是否在 `services/` 目录？
- [ ] 工具函数是否在 `utils/` 目录？
- [ ] 设置管理是否分离到独立文件？
- [ ] 目录结构是否符合推荐结构？

#### B. 调试信息管理（项目 AGENTS.md 第 7 条）
- [ ] 是否所有调试输出都受 `debugMode` 开关控制？
- [ ] 是否存在裸露的 `console.log/warn/error` 调用（不受开关控制）？
- [ ] 日志服务是否实现了级别过滤机制？

#### C. TypeScript 规范（项目 AGENTS.md 第 8 条 + 全局 AGENTS.md 编码规范）
- [ ] 是否全部使用 TypeScript？
- [ ] 接口和类型是否明确定义？
- [ ] 是否使用 Obsidian 官方类型定义？
- [ ] 命名是否符合规范（PascalCase 类名、camelCase 变量/函数、UPPER_SNAKE_CASE 常量）？
- [ ] 是否使用 `as` 而非尖括号进行类型断言？

#### D. 依赖传递（项目 AGENTS.md 第 9 条）
- [ ] 是否通过构造函数或方法参数传递依赖？
- [ ] 是否存在模块间全局变量？
- [ ] 是否使用依赖注入模式？

#### E. 错误处理（项目 AGENTS.md 第 10 条 + 全局 AGENTS.md 异常处理标准）
- [ ] 异步操作是否包含错误处理？
- [ ] 用户操作失败时是否使用 `new Notice()` 提供反馈？
- [ ] 是否存在空的 `catch` 块？
- [ ] 错误信息是否清晰具体？

#### F. 双语支持（项目 AGENTS.md 第 13.1 条）
- [ ] 所有面向用户的文案是否同时提供中文和英文？
- [ ] i18n 资源文件中中英文键值是否完整对应？
- [ ] 默认语言是否为中文？
- [ ] 是否有硬编码的用户可见文本未走 i18n？

#### G. 性能要求（项目 AGENTS.md 第 12 条）
- [ ] `onload()` 中是否有耗时操作？
- [ ] 大量数据处理是否使用异步方式？
- [ ] 事件监听器和定时器是否及时清理？

#### H. 兼容性（项目 AGENTS.md 第 13 条）
- [ ] 是否使用官方 API？是否访问了私有属性？
- [ ] 代码是否兼容当前稳定版 Obsidian？

#### I. 代码格式与风格（全局 AGENTS.md 代码格式化标准）
- [ ] 缩进是否为 2 个空格？
- [ ] 行长度是否不超过 100 个字符？
- [ ] 花括号是否在声明同一行？
- [ ] 文件末尾是否有空行？
- [ ] 导入顺序是否符合"标准库 → 第三方库 → 项目内部模块"？

#### J. 代码规模（全局 AGENTS.md 500 行限制）
- [ ] 单个文件/类/函数是否超过 500 行？

#### K. 跨平台兼容（项目 AGENTS.md 第 5-6 条）
- [ ] 路径处理是否使用 `path.join()` / `path.resolve()`？
- [ ] 是否存在硬编码的路径分隔符？
- [ ] 是否显式指定 `utf8` 编码？
- [ ] 是否依赖仅限某一平台的特性？

#### L. 安全规范
- [ ] 是否存在 XSS 风险？
- [ ] 敏感信息是否得到保护？
- [ ] 是否避免了注入攻击？

#### M. 构建与测试约束（项目 AGENTS.md 第 4、14 条）
- [ ] 是否禁止了浏览器测试代码？
- [ ] 是否禁止手动配置 sourcemap？

### 第三阶段：生成审查报告

审查完成后，在控制台或对话中输出一份摘要报告，格式如下：

```markdown
# OpenChat 代码合规审查报告

**审查时间**：YYYY-MM-DD HH:mm:ss
**审查类型**：初始审查 / 增量审查（基于 audit-report-YYYYMMDD-HHmmss.md）
**审查文件数量**：X 个
**跳过文件数量**：Y 个（上轮通过且未修改）

## 审查结果摘要

| 文件 | 状态 | 问题数 | 关键问题 |
|------|------|--------|----------|
| src/main.ts | ✅ 通过 | 0 | — |
| src/services/loggerService.ts | ❌ 不通过 | 2 | 缺少级别过滤 |
| ... | ... | ... | ... |

## 详细问题列表

### 文件：src/services/loggerService.ts
1. **[B-调试信息管理]** 缺少 debugLevel 级别过滤机制（规则来源：项目 AGENTS.md 第 7 条）
2. **[描述]** 问题描述（规则来源：xxx）

## 统计
- 通过文件：X 个
- 不通过文件：Y 个
- 总问题数：Z 个
```

### 第四阶段：修复

1. 根据报告中的每个问题，逐一修复代码
2. 修复必须遵循以下原则（来自全局 AGENTS.md）：
   - **向下兼容**：不破坏现有功能
   - **原作者视角**：保持代码风格一致
   - **KISS 原则**：选择最简单有效的方案
   - **最小修改**：只修复问题本身，不做额外重构
3. 每修复一个问题后，重新检查该文件确认问题已解决且未引入新问题
4. 修复完成后，重新输出一份更新后的审查报告确认所有问题已解决

### 第五阶段：生成锁定文档

在 `audit-reports/` 目录下创建锁定文档，文件名格式：

```
audit-report-YYYYMMDD-HHmmss.md
```

例如：`audit-report-20260323-143052.md`

锁定文档内容模板：

```markdown
# OpenChat 代码合规审查锁定文档

## 基本信息
- **审查时间**：YYYY-MM-DD HH:mm:ss
- **审查类型**：初始审查 / 增量审查
- **基于上一轮文档**：audit-report-XXXXXXXX-XXXXXX.md（若为增量审查）
- **审查前 Git 版本**：<commit hash>
- **审查后 Git 版本**：<提交后填写>

## 规则来源
- 全局规则：C:\Users\Study_Superior\.Codex\AGENTS.md
- 项目规则：AGENTS.md

## 审查范围

### 本轮审查文件（X 个）
| 文件路径 | 审查结果 | 修复前问题数 | 修复后状态 |
|----------|----------|-------------|-----------|
| src/main.ts | ✅ 通过 | 0 | 无需修复 |
| src/services/loggerService.ts | ✅ 已修复 | 2 | 全部修复 |

### 跳过文件（Y 个，上轮通过且未修改）
| 文件路径 | 上轮审查时间 | 上轮结果 |
|----------|-------------|---------|
| src/utils/helpers.ts | 2026-03-22 | ✅ 通过 |

## 修复记录
### 1. src/services/loggerService.ts
- **问题**：缺少 debugLevel 级别过滤机制
- **规则来源**：项目 AGENTS.md 第 7 条
- **修复方式**：添加 shouldLog 方法和级别比较逻辑
- **修改行数**：+15 / -3

## 当前状态
所有审查文件均符合规则约束。

## Git 版本记录
- 审查后提交版本：<git commit 后填写>
```

### 第六阶段：Git 提交与推送

1. 将所有修改的代码文件和新创建的锁定文档添加到 Git 暂存区
2. 使用以下格式提交：
   ```
   chore: code compliance audit YYYY-MM-DD

   - Audited X files against AGENTS.md rules
   - Fixed Y compliance issues
   - Generated audit lock document: audit-report-YYYYMMDD-HHmmss.md
   ```
3. 推送到远程仓库
4. 获取提交的完整 commit hash，回填到锁定文档中 `审查后 Git 版本` 字段
5. 再次提交锁定文档的版本号更新：
   ```
   chore: update audit report with commit hash
   ```
6. 再次推送

## 检查清单

每次执行审查前，确认以下事项：
- [ ] 已读取全局 AGENTS.md
- [ ] 已读取项目 AGENTS.md
- [ ] 已检查 `audit-reports/` 目录中的上一轮文档
- [ ] 已通过 git diff 确定增量变更范围
- [ ] 工作区无未提交的非审查相关变更（若有，提醒用户先处理）

## 重要约束

- **不要**审查 `node_modules/`、`dist/`、`openchat/`（构建产物）目录下的文件
- **不要**审查 `__mocks__/`、`*.test.ts` 测试文件的业务逻辑合规性（但需确认测试文件中不含浏览器测试代码）
- **不要**修改 `manifest.json`、`package.json`、`tsconfig.json` 等配置文件，除非规则明确要求
- 审查范围为 `src/` 下的所有 `.ts` 和 `.tsx` 文件，以及项目根目录下的构建脚本 `scripts/` 目录
- 锁定文档文件名必须唯一（含精确到秒的时间戳）
- 修复代码时必须确保 `npm run build` 仍然能通过
