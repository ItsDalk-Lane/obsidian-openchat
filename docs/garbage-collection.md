# docs/garbage-collection.md — 垃圾回收流程

## 为什么需要垃圾回收

AI 会忠实地复制仓库中已有的模式——包括不好的模式。
随时间推移，这必然导致漂移：

- 文档与代码脱节
- 命名规范出现分歧
- 死代码积累
- 架构边界被微妙地侵蚀

技术债是高利贷：持续小额偿还远优于让它累积到爆发。

## 清理检查项

### 文档一致性

- [ ] docs/architecture.md 中的模块依赖图与实际代码一致
- [ ] 每个域的 spec.md 与实际行为一致
- [ ] docs/quality-grades.md 评分已更新
- [ ] 没有引用不存在的文件或函数的文档

### 架构合规

- [ ] `npm run lint:arch` 零违规
- [ ] `npm run lint:taste` 零违规
- [ ] 没有绕过 Providers 直接使用 Obsidian API 的代码
- [ ] 没有域间直接导入

### 代码健康

- [ ] 没有未使用的导出（dead exports）
- [ ] 没有空文件或只有注释的文件
- [ ] 所有 TODO 都关联了具体的计划
- [ ] 没有重复的类型定义（应该在一个地方定义）

### 测试健康

- [ ] 所有测试通过
- [ ] 新增的导出函数都有对应测试
- [ ] 没有被注释掉的测试
- [ ] 没有依赖执行顺序的测试

### Obsidian 插件专项检查

- [ ] `onload()` 中无耗时同步操作
- [ ] 所有事件监听器已绑定到插件生命周期或在 `onunload()` 中清理
- [ ] 无内存泄漏风险（无孤立定时器、无未清理的 DOM 引用）
- [ ] `manifest.json` 的 `minAppVersion` 与实际 API 使用一致
- [ ] 未使用任何私有/未文档化 Obsidian API
- [ ] 所有用户可见文本已提供中英文双语版本（详见 `docs/i18n.md`）
- [ ] 默认语言仍为中文
- [ ] 跨平台检查通过（详见 `docs/cross-platform.md` 检查清单）

## 执行频率

- 每次完成一个功能后：运行完整清理检查
- 定期：扫描文档新鲜度和架构违规
- 每次发现 AI 犯同一类错误时：更新黄金原则，补强 linter
