# spec.md 编写指南

> 每个业务域目录下必须包含一个 `<domain>.spec.md` 文件，
> 用于描述该域的行为规格。它是该域的"行为契约"，也是 AI
> 理解上下文、生成代码和编写测试的首要参考。

## 定位

- spec.md 描述的是 **"做什么"（What）**，不是"怎么做（How）"
- 它不是代码注释的替代品，而是高于代码的行为定义
- AI 接到任务时，必须先读对应域的 spec.md 理解上下文
- 行为变更时，必须同步更新 spec.md（工作流程第 6 步）

## 存放位置

```text
src/domains/<domain-name>/
├── types.ts
├── config.ts
├── service.ts
├── ui.ts
├── <domain>.test.ts
└── <domain>.spec.md     ← 在这里
```

## 模板

```markdown
# <域名称> 行为规格

## 概述

一句话描述该域的职责和存在意义。

## 核心行为

### 行为 1：<动词短语描述>

- **触发条件**：用户执行 XX / 系统事件 YY
- **预期结果**：产生 ZZ 效果
- **边界情况**：
  - 当 A 不存在时 → 显示 Notice 提示
  - 当 B 超过阈值时 → 截断并警告

### 行为 2：<动词短语描述>

- **触发条件**：...
- **预期结果**：...
- **边界情况**：...

## 不做什么（显式排除）

- 该域**不负责** XX（由 <其他域> 处理）
- 该域**不直接访问** YY API（通过 Provider 注入）

## 依赖

- 依赖的 Provider：`SettingsProvider`, `ObsidianAPIProvider`
- 依赖的其他域：无 / `<domain-name>`（仅通过事件通信）

## 用户可见文案

| 场景 | 中文 | English |
|---|---|---|
| 成功提示 | 操作已完成 | Operation completed |
| 错误提示 | 处理失败：{reason} | Processing failed: {reason} |

## 变更历史

| 日期 | 变更内容 | 原因 |
|---|---|---|
| YYYY-MM-DD | 初始版本 | 域创建 |
```

## 何时创建

- **新建域时**：必须同时创建 spec.md，即使只有概述和 1-2 个核心行为
- 禁止出现"先写代码后补 spec"的情况——spec 先行，代码跟随

## 何时更新

- 新增行为 → 添加行为条目
- 修改现有行为的触发条件或预期结果 → 更新对应条目
- 删除行为 → 从 spec 中移除并记录变更历史
- 修改用户可见文案 → 同步更新文案表格

## 与测试的关系

- spec.md 中的每个"核心行为"条目应该能对应到至少一个测试用例
- 测试文件 `<domain>.test.ts` 的 `describe` 块建议与 spec.md 的
  行为条目标题保持一致，便于双向追溯

## 检查清单

- [ ] 每个域目录下都有 spec.md
- [ ] spec.md 的行为条目覆盖了该域的所有用户可感知功能
- [ ] "不做什么"区域明确划定了职责边界
- [ ] 用户可见文案表格包含中英文双语
- [ ] 变更历史记录了最近一次修改
